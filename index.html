The Problem: Manual Reporting Bottlenecks

While supporting purchasing operations at Digital Dynamics Inc., I noticed that buyers were spending hours each week manually compiling open order reports for dozens of suppliers. Each report required pulling live data from SYSPRO, cleaning it in Excel, and emailing supplier updates individually. The process was repetitive, error-prone, and unsustainable as volume scaled.

End-to-End Process Overview

The system flows through three stages: custom SQL logic in SYSPRO, a structured Excel extract, and a Power Automate cloud flow that handles report generation and delivery automatically.

My Approach

I took ownership of the solution end to end. That meant writing the custom SQL query, building the Excel architecture that powers the report, and creating the Power Automate flow that distributes supplier-specific files on a recurring schedule.

Step 1 — Building the Core SQL Logic

I wrote a custom SQL query from scratch that identifies all open purchase orders with outstanding quantities, de-duplicates based on job and line, resolves the appropriate stock code description, flags late deliveries, and extracts supplier, warehouse, and due date data into one structured output.

Custom SQL query powering the automated PO OOR report

oor_report_query.sql 
-- Custom SQL logic powering the automated OOR system
WITH DeduplicatedBI AS (
	SELECT
		SourcePurchaseOrder, 
        SourcePurchaseOrderLine, 
        SourceJob,
        DateNeeded,
        ROW_NUMBER() OVER (
            PARTITION BY SourcePurchaseOrder, SourcePurchaseOrderLine, SourceJob 
            ORDER BY DateNeeded ASC) AS RowNum
    FROM BI_Material_PullIn), -- Keeps earliest DateNeeded
UseableStockCode AS (
	SELECT
		PurchaseOrder,
		Line,
	CASE 
		WHEN MStockCode IN ('00LABOR') THEN LTRIM(RTRIM(MStockDes))
		ELSE MStockCode
	END AS UseableStockCode
	FROM PorMasterDetail AS PMD)
SELECT
	PMH.Supplier,
	CAST(PMD.PurchaseOrder AS BIGINT) AS PurchaseOrder,  
	PMD.MStockCode AS StockCode,
	PMD.MStockDes AS StockCodeDescription,
	UseableStockCode.UseableStockCode AS UseableStockCode, -- Logic correct after testing
	COALESCE(InvMaster.Version, '') AS StockCodeRev,
	PMD.Line,
	PMD.MWarehouse AS Warehouse,
	CAST(PMD.MOrderQty AS INT) AS OrderQty,
	CAST(PMD.MReceivedQty AS INT) AS ReceivedQty,
	CAST((PMD.MOrderQty - PMD.MReceivedQty) AS INT) AS OutstandingQty,
	PMH.OrderEntryDate AS OrderEntryDate, -- DOMO / EXCEL / POWER BI can convert dates after uploading
	PMD.MOrigDueDate AS OrigDueDate, -- DOMO / EXCEL / POWER BI can convert dates after uploading
	PMD.MLatestDueDate AS LatestDueDate, -- DOMO / EXCEL / POWER BI can convert dates after uploading
	CASE
		WHEN PMD.MLatestDueDate < CAST(GETDATE() AS DATE) THEN 'Past Due'
		ELSE ''
	END AS PastDueFlag,
	CASE
		WHEN DeduplicatedBI.DateNeeded IS NULL THEN ''
		ELSE CONVERT(varchar, DeduplicatedBI.DateNeeded, 101)
	END AS DateNeeded,
	PMD.MJob AS Job
FROM
	PorMasterDetail AS PMD
INNER JOIN PorMasterHdr AS PMH
	ON PMD.PurchaseOrder = PMH.PurchaseOrder
LEFT JOIN DeduplicatedBI
	ON PMD.PurchaseOrder = DeduplicatedBI.SourcePurchaseOrder
	AND PMD.Line = DeduplicatedBI.SourcePurchaseOrderLine
	AND DeduplicatedBI.RowNum = 1
INNER JOIN UseableStockCode
	ON PMD.PurchaseOrder = UseableStockCode.PurchaseOrder
	AND PMD.Line = UseableStockCode.Line
LEFT JOIN InvMaster
	ON UseableStockCode.UseableStockCode = InvMaster.StockCode
WHERE
	(PMD.MOrderQty - PMD.MReceivedQty) > 0
	AND (PMH.OrderStatus <> '*')
	AND (PMD.MCompleteFlag <> 'Y')
	AND (PMD.MJob = DeduplicatedBI.SourceJob OR DeduplicatedBI.SourceJob IS NULL)
ORDER BY
	PMD.MLatestDueDate

The Problem: "Available" Inventory Was Not Always Easy to Trust

DDI relied heavily on tribal knowledge inside SYSPRO, which made serialized availability harder to trust than it should have been. Teams could see inventory in the system, but they still lacked a clean way to validate what finished goods were actually usable by warehouse across 40+ locations. Operations needed faster visibility, and finance needed better support for consignment-related inventory assignment so stock could be decremented correctly in SYSPRO.

This was especially important once inventory moved through the consignment workflow. It was not enough to know that a part existed somewhere in the system. The team needed a clearer way to understand what was unallocated, what warehouse it sat in, and what serial-level detail supported the next step.

What the Tool Needed to Solve

  • Show which specific serial numbers were available or unavailable at the detail level
  • Summarize total unallocated FGI by stock code and warehouse for faster review
  • Support operational checks across standard and consignment-relevant warehouses such as 4F, K1, L1, and M1
  • Give finance a clearer starting point when assigning FGI within the consignment process so inventory could be decremented correctly in SYSPRO

End-to-End Logic Overview

The solution used SQL to create two practical views from InvSerialHead: a serial-level detail extract and a grouped Total Unallocated FGI summary. Those outputs were then structured in Excel so users could start from the high-level warehouse summary and drill into the underlying serial detail only when they needed it.

SQL Layer

The SQL here should be read as representative logic that lines up more closely with the full project story. The underlying work still relied on Excel views and business workflow decisions, but this version better reflects how serial-level availability, grouped unallocated FGI, and open order context fit together inside the broader SYSPRO process.

The query below is a representative extract used for that grouped visibility layer. It helped surface unallocated FGI by stock code and warehouse, while the full project also depended on the Excel views and downstream process decisions built around those outputs.

Representative SQL that combines serial-level availability, grouped unallocated FGI, and open sales-order context for the workflow described above

total_unallocated_fgi.sql 
-- Representative SQL for the serial availability project
-- This combines three layers that mattered in practice:
-- 1) serial-level visibility from InvSerialHead
-- 2) grouped unallocated FGI by stock code and warehouse
-- 3) open sales-order context from SorMaster / SorDetail

WITH SerialDetail AS (
    SELECT
        IH.StockCode,
        IH.Warehouse,
        IH.Serial,
        IH.SerialDescription,
        CAST(IH.QtyReceived AS INT) AS QtyReceived,
        CAST(IH.QtyAvailable AS INT) AS QtyAvailable,
        CAST(IH.QtyOnHand AS INT) AS QtyOnHand,
        CASE
            WHEN IH.QtyAvailable > 0 THEN 'Available'
            ELSE 'Unavailable'
        END AS SerialStatus
    FROM InvSerialHead AS IH
    WHERE IH.Warehouse IN ('4F', '4X', '4S', '4V', '4Z', 'K1', 'L1', 'M1')
),
UnallocatedFGI AS (
    SELECT
        SD.StockCode,
        SD.Warehouse,
        MAX(SD.SerialDescription) AS SerialDescription,
        CAST(SUM(SD.QtyAvailable) AS INT) AS TotalQtyAvailable,
        COUNT(*) AS AvailableSerialCount
    FROM SerialDetail AS SD
    WHERE SD.QtyAvailable > 0
    GROUP BY SD.StockCode, SD.Warehouse
),
OpenSalesOrders AS (
    SELECT
        SM.Customer,
        SM.CustomerName,
        '0' + LTRIM(CAST(CAST(SM.SalesOrder AS BIGINT) AS VARCHAR)) AS SalesOrder,
        SM.CustomerPoNumber,
        SD.MStockCode AS StockCode,
        SD.MStockDes AS StockCodeDescription,
        CAST(SD.MOrderQty AS INT) AS OrderQty,
        CAST(SD.MBackOrderQty AS INT) AS BackOrderQty,
        CAST(SD.MShipQty AS INT) AS ShipQty,
        FORMAT(SD.MLineShipDate, 'MM/dd/yyyy') AS ShipDate,
        SD.MWarehouse AS Warehouse,
        SD.SalesOrderLine,
        CASE
            WHEN SD.MWarehouse IN ('K1', 'L1', 'M1') THEN 'Consignment Warehouse'
            WHEN SD.MWarehouse = '4F' THEN 'Standard FGI Warehouse'
            ELSE 'Other Warehouse'
        END AS WarehouseRole
    FROM SorMaster AS SM
    INNER JOIN SorDetail AS SD
        ON SM.SalesOrder = SD.SalesOrder
    WHERE CAST(SM.OrderStatus AS VARCHAR) NOT IN ('9', 'F', 'S', '\', '*', '8')
      AND SD.LineType NOT IN ('6', '5', '7')
      AND NOT (CAST(SD.MBackOrderQty AS INT) = 0 AND CAST(SD.MShipQty AS INT) = 0)
)
SELECT
    OSO.Customer,
    OSO.CustomerName,
    OSO.SalesOrder,
    OSO.CustomerPoNumber,
    OSO.StockCode,
    OSO.StockCodeDescription,
    OSO.Warehouse,
    OSO.WarehouseRole,
    OSO.OrderQty,
    OSO.BackOrderQty,
    OSO.ShipQty,
    OSO.ShipDate,
    ISNULL(UFGI.TotalQtyAvailable, 0) AS UnallocatedFGIQty,
    ISNULL(UFGI.AvailableSerialCount, 0) AS AvailableSerialCount,
    CASE
        WHEN ISNULL(UFGI.TotalQtyAvailable, 0) >= OSO.BackOrderQty THEN 'Inventory available to review'
        WHEN ISNULL(UFGI.TotalQtyAvailable, 0) > 0 THEN 'Partial inventory available'
        ELSE 'No unallocated serial inventory found'
    END AS AvailabilitySummary
FROM OpenSalesOrders AS OSO
LEFT JOIN UnallocatedFGI AS UFGI
    ON OSO.StockCode = UFGI.StockCode
   AND OSO.Warehouse = UFGI.Warehouse
ORDER BY OSO.Warehouse, OSO.CustomerName, OSO.StockCode, OSO.SalesOrderLine;

What I Built

I built a burn down model in Excel, supported by SQL, SYSPRO, and DOMO extracts, to estimate projected ending inventory at DDI. The model combined on hand inventory, open backlog, open purchase orders, and outstanding jobs to show what value would burn down through end of Q3'25 and what would remain.

How the Model Worked

The model focused specifically on target top level assemblies that were driving excess inventory exposure. To make the output decision useful, I narrowed the analysis to the warehouses, build types, and time horizon that mattered most.

  • Pulled open jobs from SYSPRO to capture unfinished production activity due within the burn down window and summed remaining quantity from QtyToMake minus QtyManufactured.
  • Used open backlog to represent customer demand still needing fulfillment within the modeled horizon.
  • Used open purchase orders to estimate inbound raw parts and finished goods, including finished goods returning from overseas manufacturer SPARV.
  • Used part master data to identify on hand and on order quantities in the key 4F and 4X inventory buckets.
  • Focused the analysis on target top level assemblies, split between USA box builds, overseas box builds, and BIB/PWA groupings.
  • Layered in tier classifications, OBS and LTB status, and unit cost data to translate quantity movement into inventory value impact.

Why Warehouse and Build Type Mattered

Warehouse logic mattered because not all inventory served the same purpose. I separated the analysis to focus on the buckets that actually affected DDI's exposure, especially 4F and 4X. I also split the model by build type because USA box builds, overseas box builds, and BIB/PWA groupings behaved differently in demand, inbound supply, and remaining value.

What the Analysis Showed

The final output translated multiple operational data sources into an executive level inventory view. Instead of only showing current stock, the model estimated projected ending inventory and value reduction through end of Q3'25. That made it easier to see what would burn down, what would remain, and where holding exposure was still meaningful.

Why It Mattered

This analysis gave the business a more realistic view of inventory exposure than a static on hand snapshot. It connected inventory to projected demand, production activity, inbound supply, and dollar impact, making it more useful for evaluating future holding reduction and inventory decisions at DDI.

Project Overview

This Power BI project page is meant to show the type of executive dashboard experience I want represented in my portfolio. Instead of dumping raw charts onto a page, the goal is to organize the most important business signals into a layout leadership can read quickly and actually use.

What this dashboard emphasizes

  • High visibility KPIs at the top for immediate status checks
  • Regional and monthly trend views that support faster pattern recognition
  • Cleaner design hierarchy so the dashboard feels decision oriented, not cluttered
  • A data model foundation that could support row level security, scheduled refreshes, and reusable DAX logic

Why it matters

Executive dashboards are only valuable when they reduce friction. The point is not just to show metrics, but to make performance easier to understand, compare, and act on. This page demonstrates the reporting style and dashboard structure I bring to leadership-facing Power BI work.
